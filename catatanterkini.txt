1. FITUR ADMIN, PERAN (ROLE), DAN SISTEM SEEDING
-----------------------------------------------------------------------------
* KENDALA AWAL:
  - Admin default tidak ter-seed jika database sudah berisi satu atau lebih user biasa.
  - Formulir pendaftaran manual admin tidak memiliki opsi peran (Role) pengguna.
  - Endpoint admin (/api/v1/admin/*) belum dilindungi pengaman otentikasi admin.
  - Risiko lockout admin jika tidak sengaja menghapus atau menonaktifkan akun sendiri.

* SOLUSI & PERUBAHAN KODE:
  - Seeding Default Admin (core/database.py):
    Logika diubah dari mengecek kekosongan tabel menjadi mengecek secara spesifik keberadaan
    email 'admin@gmail.com'. Ditambahkan penanganan 'IntegrityError' (try-except) untuk
    mencegah kegagalan startup uvicorn ketika dijalankan dalam mode multi-worker.
  - Keamanan API (api/deps.py & api/v1/admin.py):
    Menambahkan dependency 'get_current_admin' yang mengecek status 'is_admin = True'.
    Seluruh API dashboard admin kini dilindungi dengan dependensi ini.
  - Fitur Saling Menghapus Admin:
    Admin diperbolehkan menghapus admin lain. Namun, ditambahkan proteksi pengecekan
    (user_id == admin.id) pada aksi DELETE, DEACTIVATE, dan FORCE LOGOUT agar admin
    tidak bisa melukai akunnya sendiri (menghindari lockout).
  - Formulir Tambah Admin Manual (static/admin.html):
    Menambahkan select-dropdown 'Role Pengguna' (USER / ADMIN) dan merapikan layout
    formulir pendaftaran manual menjadi 5 kolom grid di desktop.

2. PENANGANAN KENDALA TEKNIS VPS AAPANEL & NGINX
-----------------------------------------------------------------------------

A. ERR_HTTP2_PROTOCOL_ERROR / TypeError: Network Error saat Video Render
   * GEJALA: Progres bar mendadak macet, muncul "TypeError: network error" di PC
     atau "load failed" di HP saat video sedang dianalisis oleh AI.
   * PENYEBAB: Proses AI analisis & FFmpeg rendering memakan waktu 15-40 detik. Selama
     rentang waktu ini, tidak ada bytes data baru yang dikirim di jaringan TCP.
     Nginx, Cloudflare, atau browser menganggap koneksi mati dan memutusnya secara paksa.
   * SOLUSI: Menerapkan mekanisme 'Heartbeat / Keep-Alive' di backend (api/v1/generate.py).
     Jika tidak ada update baru dari Redis selama 5 detik, backend akan mengirimkan
     SSE comment `: keepalive\n\n` secara terus menerus agar koneksi tetap dianggap aktif.

B. HTMX 401 Unauthorized pada Load Pertama
   * GEJALA: Saat admin membuka dashboard, tabel stats, user, dan log memunculkan
     pesan merah "Response Status Error Code 401" padahal admin sudah login.
   * PENYEBAB: Pustaka HTMX dimuat sebelum event listener 'htmx:configRequest' didaftarkan.
     HTMX langsung mengirim request onload tanpa membawa token JWT Bearer.
   * SOLUSI: Mengatur ulang urutan script di kepala halaman <head> (static/admin.html).
     Script pendaftaran token JWT HTMX dan client-side redirect guard dipindahkan ke
     paling atas sebelum script CDN HTMX dimuat.

C. Address Already in Use (Errno 98) / 502 Bad Gateway
   * GEJALA: Nginx menampilkan 502 Bad Gateway, dan log supervisor/terminal memunculkan
     pesan "Address already in use".
   * PENYEBAB: Proses uvicorn lama masih aktif berjalan di background dan mengunci port 8000.
     Hal ini biasa terjadi ketika reload Supervisor tidak sempurna mematikan proses lama.
   * SOLUSI: Matikan paksa proses lama yang mengunci port 8000 di terminal VPS:
     $ fuser -k 8000/tcp
     atau
     $ kill -9 $(lsof -t -i:8000)
     Setelah itu jalankan `$ supervisorctl reload` untuk menyalakan kembali.

D. ValueError: Invalid isoformat string: 'monthly'
   * GEJALA: Aplikasi gagal me-load daftar user di dashboard admin.
   * PENYEBAB: Nilai string teks 'monthly' masuk ke dalam kolom bertipe tanggal 'expired_at'
     (kemungkinan akibat bug input manual lama sebelum diperbaiki).
   * SOLUSI: Membuat skrip pembersih database agnostic 'db_clean.py'. Skrip ini membaca
     DATABASE_URL dari file .env secara otomatis dan membersihkan nilai 'monthly'
     pada kolom 'expired_at' menjadi NULL secara aman.
     Cara menjalankan di VPS:
     $ source venv/bin/activate
     $ python3 db_clean.py

E. FFmpeg Timeout (300 seconds limit exceeded) / Endless Keep-Alive
   * GEJALA: Proses video terhenti pada tahap render FFmpeg (Step C) dengan status `: keepalive`
     secara terus-menerus hingga akhirnya gagal setelah 300 detik dengan pesan timeout.
   * PENYEBAB: Penggunaan opsi `-stream_loop -1` (loop video tanpa batas) yang digabungkan dengan
     flag `-shortest` dan filter audio `amix` memicu bug pada beberapa versi FFmpeg. FFmpeg
     gagal mendeteksi akhir (EOF) dari stream audio narasi, menyebabkan proses rendering looping
     terus-menerus tanpa henti hingga melampaui batas timeout 300 detik. Namun, jika `-stream_loop`
     dihapus sepenuhnya, FFmpeg akan hang/crash ketika input video kehabisan frame sebelum mencapai
     durasi output target `-t` (misal karena perbedaan presisi durasi audio/video).
   * SOLUSI: Menggunakan kombinasi `-stream_loop -1` pada input video agar frame video selalu tersedia
     tanpa batas (menghindari stall/hang akibat kehabisan frame), dan menerapkan opsi durasi keras
     `-t <video_duration>` pada output akhir di `core/pipeline.py`. Dengan demikian, FFmpeg dijamin
     selesai tepat waktu mengikuti panjang video input asli secara presisi (tidak melebihi durasi video
     dan tidak mengikuti durasi narasi), serta 100% bebas dari kendala hang/timeout/stalling.

3. MIGRASI DATABASE KE MYSQL AAPANEL (100% SUKSES)
-----------------------------------------------------------------------------
* KENAPA BERMIGRASI KE MYSQL?
  - SQLite kurang optimal untuk ribuan pengguna konkuren karena mengunci file database
    saat melakukan operasi tulis (menyebabkan database locked).
  - MySQL bawaan aaPanel sangat handal menangani ribuan concurrent write dan stabil.
  - Arsitektur backend kita sudah dinamis (dialect-aware). Cukup ubah konfigurasi .env
    di VPS, sistem akan otomatis melakukan migrasi skema tabel tanpa mengubah kode Python.

* FORMAT DATABASE_URL MYSQL (di VPS .env):
  DATABASE_URL="mysql+aiomysql://username_db:password_db@127.0.0.1:3306/nama_db"

* CARA RESTART SERVER SETELAH UBAH .ENV:
  $ source venv/bin/activate
  $ fuser -k 8000/tcp
  $ supervisorctl reload

5. CARA MELIHAT LOG ERROR DI VPS (MENGGUNAKAN PUTTY / TERMINAL SSH)
-----------------------------------------------------------------------------
Jika terjadi kendala (seperti 502 Bad Gateway atau error pemrosesan video), Anda
WAJIB login ke VPS menggunakan Putty atau Terminal SSH, lalu jalankan perintah
berikut untuk memantau log error secara langsung (real-time):

A. MEMANTAU LOG EROR DARI SUPERVISOR (BACKGROUND RUN)
   Untuk memantau aktivitas & error backend yang sedang berjalan di background:
   $ supervisorctl tail -f video_saas
   *(Catatan: Ganti 'video_saas' dengan nama tugas/task yang Anda buat di Supervisor aaPanel)*

B. MEMANTAU LOG ERROR NGINX (WEB SERVER)
   Untuk melihat jika request diblokir oleh Nginx (misal karena ukuran video kebesaran):
   $ tail -n 50 -f /www/wwwlogs/ai.agomart.com.error.log

C. MENJALANKAN MANUAL UNTUK MENAMPILKAN TRACEBACK ASLI (SANGAT DIREKOMENDASIKAN)
   Jika Supervisor error terus-menerus dan Anda ingin melihat traceback asli dari Python:
   1. Hentikan sementara semua proses background:
      $ supervisorctl stop all
   2. Bersihkan port 8000 yang terkunci:
      $ fuser -k 8000/tcp
   3. Masuk ke Virtual Environment & jalankan secara manual:
      $ source venv/bin/activate
      $ uvicorn main:app --host 0.0.0.0 --port 8000
   
   *(Semua error asli Python, koneksi database, dan log Qwen/FFmpeg akan langsung 
     tercetak di layar Putty secara instan saat Anda mencoba mengakses website!)*

6. PANDUAN DEPLOYMENT MENGGUNAKAN GIT
-----------------------------------------------------------------------------
Berikut adalah panduan lengkap cara melakukan sinkronisasi kode dari komputer lokal (PC/Mac) Anda ke server VPS aaPanel menggunakan Git.

A. CARA GIT COMMIT DARI PC ATAU MAC (KOMPUTER LOKAL)
   Lakukan langkah ini setiap kali Anda selesai melakukan perubahan kode di komputer Anda:
   
   1. Buka Terminal (Mac) atau Command Prompt/Git Bash (Windows) di folder project.
   2. Pastikan posisi Anda berada di branch utama (misal: 'main'):
      $ git branch
   3. Tambahkan semua file yang mengalami perubahan ke dalam staging area:
      $ git add .
   4. Buat catatan commit (simpan riwayat perubahan kode):
      $ git commit -m "feat: tambah menu convert video size dan sidebar premium"
   5. Kirim (upload) perubahan kode lokal tersebut ke repository online (GitHub/GitLab):
      $ git push origin main

B. CARA GIT PULL KE VPS AAPANEL (UPDATE KODE DI SERVER)
   Lakukan langkah ini untuk menarik pembaruan kode terbaru yang baru saja Anda push:
   
   1. Masuk ke VPS Anda menggunakan Terminal SSH atau Putty:
      $ ssh root@154.26.139.162
   2. Masuk ke direktori web/project Anda di VPS (sesuaikan nama folder web):
      $ cd /www/wwwroot/ai.agomart.com
   3. Tarik (download) perubahan kode dari GitHub/GitLab:
      $ git pull origin main
      
      *(Catatan: Jika muncul error conflict akibat ada file yang tidak sengaja terubah langsung di VPS,
      Anda dapat membatalkan perubahan lokal VPS tersebut dengan perintah: `$ git checkout -- .` 
      lalu ulangi langkah `$ git pull origin main`)*
      
   4. Terapkan perubahan dengan me-restart server backend dan Celery worker:
      $ supervisorctl restart all
      
      *atau matikan paksa port 8000 dan muat ulang Supervisor:*
      $ fuser -k 8000/tcp
      $ supervisorctl reload
